# Contributing Guidelines

We welcome contributions from the community. Please read through these guidelines before contributing.

### Getting Started

To contribute to this project, start by `forking` the `repository on Github` . `Clone` your `fork` locally and create a new branch for your changes. Make sure to keep your `fork synchronized` with the upstream `repository` to avoid conflicts.

Before starting work on a significant change, please check the existing issues and pull requests to avoid duplication of effort. If you are planning a major feature or change, consider opening an issue first to discuss it with the `maintainers`.

### Types of `Contributions`

We accept various types of contributions including:

##### `< Code Contributions />`

Bug fixes, new features, performance improvements, and refactoring existing code. All code contributions should follow the projects coding standards and include appropriate tests.

##### `< Documentation />`

Improvements to README files, code comments, API Documentation , tutorials, and examples. Clear documentation helps everyone understand and use the project better.

##### `< Bug Reports />`

Detailed bug reports with reproducible steps help us identify and fix issues quickly. Include information about your environment and any error messages.

##### `< Feature Requests />`

Suggestions for new features or enhancements. Explain the use case and how it would benefit the project and its users.

### Reporting Issues

When reporting issues on GitHub, please provide as much detail as possible to help us understand and reproduce the problem. A good issue report includes:

##### `Clear Title`

A concise description of the issue.

##### `Description`

Detailed explanation of what went wrong and what you expected to happen.

##### `Steps to Reproduce`

 Numbered steps that allow us to recreate the issue.

##### `Environment`

Your operating system, browser version, Node.js version, and any other relevant details.

##### `Screenshots or Logs`

- If applicable, include screenshots or error logs that demonstrate the problem.

- Before creating a new issue, please search existing issues to see if someone has already reported the same problem.

##### For Example

```
Title: Analysis API returns CORS error when called from browser

Description:
Expected XYZ to work, but encountered ABC.

Steps:
1. Start Next.js dev server
2. Trigger analysis
3. Observe 403 on OPTIONS request

Environment:
OS: Windows 11
Browser: Chrome 121
Node: 18.17.1
Next.js: 14.1.0
```

### Pull Request Process

When submitting a pull request, follow these steps:

- Create a new branch from the main branch with a descriptive name.
- Make your changes and commit them with clear, concise commit messages.
- Write or update tests as necessary to cover your changes.
- Ensure all tests pass and there are no linting errors.
- Update documentation to reflect any changes in functionality.
- Push your changes to your fork and submit a pull request to the main repository.
- In your pull request description, explain what changes you made and why.
- Link any relevant issues that your pull request addresses.
- A maintainer will review your pull request and may request changes or provide feedback. Please be responsive to comments and willing to make adjustments as needed.

### Code Standards

To maintain code quality and consistency across the project, please adhere to these standards:

*Follow the existing code style and formatting conventions used in the project.*

- Write clear, self-documenting code with meaningful variable and function names.

- Add comments to explain complex logic or non-obvious decisions.

- Keep functions small and focused on a single responsibility.

- Avoid unnecessary dependencies and keep the codebase lean.

- Run linters and formatters before committing your code.

### Becoming a Contributor

- Once your pull request is merged, you will automatically be recognized as a contributor to the project. We value all contributions, whether they are small bug fixes or major features.

- Active contributors who consistently provide high-quality contributions may be invited to become maintainers with additional responsibilities and permissions in the repository.

We appreciate your time and effort in helping improve this project. Thank you for contributing!

- **If you have questions about contributing, feel free to open an issue on GitHub or reach out to the maintainers.**
